Elements vs Components

Here's a question for you. If this is called a function definition,

function add(x, y) {
  return x + y;
}

this is called a function invocation.

add(1, 2)

If this is called a component definition,

function Icon () {
  return <svg />
}

what is this called?

<Icon />

Here are some of the most common answers, all of which are incorrect.

  1. 1. Creating an instance/instantiation
  2. 2. Rendering a component
  3. 3. Evaluating a component
  4. 4. Component invocation
  5. 5. Using it 😅

The primary reason for the confusion is because there's an important but subtle step between when React evaluates your code, and the browser sees it. This makes sense. The browser can only understand JavaScript – and though JSX is a syntax extension to JavaScript, it's still not browser compatible JavaScript.

This problem isn't anything new. We've been non-browser-compatible syntax (Elm, TypeScript, CoffeeScript, etc.) to JavaScript for years.

So how exactly do we go from <Icon />, to JavaScript the browser can understand?

The answer lies in what React calls an element.

What is a React Element?

An object representation of a DOM node

This definition gives us some interesting insight into how React works under the hood. An element isn't a DOM node, instead it's an object representation of it.

Here's an example of what one looks like.

{
  type: "h1",
  props: {
    children: "Profile",
    className: "header"
  }
}

This, of course, is an object representation of a DOM node that looks like this.

<h1 class="header">
  Profile
</h1>

The way React works is it'll keep references to all of the elements in your application. If one of those references changes, React will then know where exactly in the DOM it needs to update.

In order to create a React element, you have a few options – one of them being React's jsx-runtime package.

import { jsx } from "react/jsx-runtime"


const element = jsx("h1", {
  className: "header",
  children: "Profile"
})

jsx takes in two arguments. The first is a tag name string (div, span, etc.) and the second is an object of attributes you want the element to have. In our case, a className of header with the text Profile as its contents.

What it returns is a React element as shown above. We can see this in action with the following code.

import { jsx } from "react/jsx-runtime"

export default function App () {
  return jsx("h1", {
    className: "header",
    children: "Profile"
  })
}

Notice there's no JSX – it's Just JavaScript™.

What’s interesting about learning React is that typically one of the first things you're taught are components and that components are the building blocks of React. As we can see, that's not true – elements are the true building blocks of React. Components, by definition, are just functions that optionally accept input via props, and return a React element.

In our example above, App is a component and what it returns (the invocation of jsx) is a React element.

import { jsx } from "react/jsx-runtime"


const Component = () => {
  const element = jsx("h1", {
    className: "header",
    children: "Profile"
  })


  return element
}


export default Component

And you're not only limited to creating React elements from native HTML elements (span, div, etc.). You can also pass in other React components as the first argument to jsx as well.

const element = jsx(Badge, {
  name: "Tyler McGinnis",
  handle: "tylermcginnis",
  img: "https://avatars0.githubusercontent.com/u/2933430"
})

When React sees a component as the first argument, it will recursively render that component as well as all of its children components in order to get all the React elements and the final description of the UI.

Remember the Badge component from the props lesson? Here's what it looks like using the jsx-runtime package.

img {
width: 200px;
border-radius: 50%;
}

h4 {
margin: 5px;
}

p {
margin: 5px;
}

.badge {
width: 300px;
margin: 0 auto;
border-radius: 8px;
background-color: var(--charcoal);
padding: 24px;
display: flex;
flex-direction: column;
align-items: center;
gap: 16px;
text-align: center;
}

So now the big question you've probably been wondering, what's the difference between writing JSX and using the jsx transform?

As far as the browser is concerned, nothing.

As you can imagine, creating a description of your UI with jsx function invocations isn't terribly ergonomic – which is why we have JSX. At build time, all the JSX in your app gets compiled to Javascript via jsx invocations that the browser can understand.

So now that we have some context, let's go back to our original question.

If this is called a function definition,

function add(x, y) {
  return x + y;
}

this is called a function invocation.

add(1, 2)

If this is called a component definition,

function Icon () {
  return <svg />
}

what is this called?

<Icon />

We call it creating an element, because after the JSX is compiled and jsx is invoked, that’s what we get back – an element.

In fact, all of these examples are showing different ways to create React elements, regardless of if we're using JSX or the jsx transform API.

<Hello />


<div className="container">
  Hello!
</div>


jsx(Badge, {
  name: "Tyler McGinnis",
  handle: "tylermcginnis",
  img: "https://avatars0.githubusercontent.com/u/2933430"
})
