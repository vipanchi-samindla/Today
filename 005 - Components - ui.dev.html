Components

One thing I love about React is that it feels like it's the answer to the question, What would HTML look like if it were invented today?

The primary benefit of HTML is that it allows us to create rich, structured markup in a declarative way. In contrast, the primary benefit of JavaScript is that it gives us a set of tools for managing the complexity of building an application – namely through features like the module system, scopes, closures, and anything else you'd expect from an actual programming language.

React, of course, is a lovechild of the two – giving you the declarative-ness of HTML with the feature set of JavaScript. But, what does that actually look like.

Let's say we had some HTML.

<aside>
  <h2>Authors</h2>
  <ul>
    <li>Tyler McGinnis</li>
    <li>Ben Adam</li>
    <li>Lynn Fisher</li>
  </ul>
</aside>

Now, say we wanted to re-use this HTML on multiple pages. How would we go about doing that? The most common way is with this neat trick.

🥲

How to Reuse HTML

"CMD + C" then "CMD + V"

The ole' copypasta. It works, but... it's not ideal. In JavaScript, this is already a solved problem. We can put this code in a function, then put that function in a module, then import that function anywhere we want to use it.

If we were able to take those JavaScript features and apply them to our HTML, that might look something like this.

export default function authors () {
  return <aside>
    <h2>Authors</h2>
    <ul>
      <li>Tyler McGinnis</li>
      <li>Ben Adam</li>
      <li>Lynn Fisher</li>
    </ul>
  </aside>
}

Believe it or not, this is essentially React.

I wasn't joking when I said React is the lovechild of HTML and JavaScript. If you've never seen a React component before, there's a good chance you're mildly horrified right now.

The first thing you'll notice is that a React component is just a function. Again, React loves JavaScript. The same mental model you have about functions can be directly applied to React components because, in the literal sense, they're just functions.

The only change we need to make it a true React component is to capitalize the first letter of our function.

export default function Authors () {
  return <aside>
    <h2>Authors</h2>
    <ul>
      <li>Tyler McGinnis</li>
      <li>Ben Adam</li>
      <li>Lynn Fisher</li>
    </ul>
  </aside>
}
🚨

React Components vs DOM Elements

The reason you need to capitalize the first letter of your component is so React knows it's a React component and not a normal DOM element – i.e. Footer vs footer.

React Component

HTML Element

And now, because we've encapsulated this component inside its own module, anywhere where we want to re-use it we can just import it.

import Authors from './Authors'

export default function About () {
  return <main>
    <h1>About Us</h1>
    <p>We write JavaScript and words about JavaScript.</p>
    <Authors />
  </main>
}
🧱

Using a Component

Notice that we don't invoke Authors as if it's a normal function – i.e. Authors(). Instead, we treat it as if it were an HTML element, wrapping it in angle brackets – i.e. <Authors />.

There's a lot happening under the hood between when JavaScript reads the component and when the browser paints the UI to the screen, but know that the final output is probably what you'd expect – where React replaces the component with the description of the UI the component returns.

<main>
  <h1>About Us</h1>
  <p>We write JavaScript and words about JavaScript.</p>
  <aside>
    <h2>Authors</h2>
    <ul>
      <li>Tyler McGinnis</li>
      <li>Ben Adam</li>
      <li>Lynn Fisher</li>
    </ul>
  </aside>
</main>

This is true regardless of how many components there are, or how deeply they're nested.

And since components are just functions, again, the same intuition you have about organizing functions and when to abstract logic to its own function can be directly applied to components.

If you don't yet have that intuition, following the single responsibility principle is a decent barometer for when and where to create components. Namely, components (and functions) should do just one thing.

When you follow the Single Responsibility Principle, you're likely to end up with a lot of smaller components. Just like with functions, this isn't a bad thing. However, it can overwhelm your component directory if you're not careful.

Take this component for example.

import Sidebar from './Sidebar'
import Timeline from './Timeline'
import Search from './Search'
import Trending from './Trending'
import Follow from './Follow'
import Footer from './Footer'


export default function Home () {
  return <main>
    <Sidebar />
    <Timeline />
    <Search />
    <Trending />
    <Follow />
    <Footer />
  </main>
}

It follows the single responsibility, but since we're creating a new file for each component, it's a bit messy.

Instead, here's a good rule of thumb you can follow. If you have a component that's being reused somewhere else, make it its own file. If not, just create it inside of whatever file you need it.

For example, we can refactor the code above like so – assuming Timeline, Trending, and Follow are only used inside of this Home component.

import Sidebar from './Sidebar'
import Search from './Search'
import Footer from './Footer'


function Timeline () {
  ...
}


function Trending () {
  ...
}


function Follow () {
  ...
}


export default function Home () {
  return <main>
    <Sidebar />
    <Timeline />
    <Search />
    <Trending />
    <Follow />
    <Footer />
  </main>
}

React and Pure Functions

A function is considered pure if it contains no side effects and if, given the same input, it always returns the same output.

By embracing and striving to keep your functions pure, you'll naturally increase the predictability of your program while minimizing the amount of bugs it has.

For this reason, React embraces pure functions and, as a rule, you need to make sure your components are always pure.

We'll learn all about strategies for this later on in the course, but as a sneak peek, just know that any side effects your component has should be wrapped inside of event handlers.
